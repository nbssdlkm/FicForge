# FicForge 交互细节补充设计（5个待明确点）

> 补充主PRD和补充PRD中描述不够清晰、实现时可能出歧义的交互设计。
> 创作模式为 Phase 2，本文所有设计仅针对嗑粮模式。

---

## 1. 多稿机制的完整交互流

### 1.1 怎么触发多稿

PRD 已定义草稿持久化到 `.drafts/`，但触发方式没说清。

```
第一稿：用户点 [续写] → AI 生成 → 写入 ch038_draft_A.md → 显示正文

此时界面状态：
┌──────────────────────────────────────┐
│                                      │
│          （第一稿正文）                │
│                                      │
│  草稿 1/1                            │
│  [✅ 定稿]  [🔄 换一版]  [🗑️ 丢弃]   │
└──────────────────────────────────────┘

用户点 [换一版]：
  → 弹出选项：
    "用相同指令重新生成" ← 默认选中（同参数，不同随机种子）
    "修改指令后重新生成" ← 点击后输入框激活，用户改完点发送
  → AI 生成第二稿 → 写入 ch038_draft_B.md

此时界面状态：
┌──────────────────────────────────────┐
│                                      │
│          （第二稿正文）                │
│                                      │
│  [◀ 上一稿]  草稿 2/2  [▶ 下一稿]    │
│  [✅ 定稿]  [🔄 换一版]  [🗑️ 全部丢弃]│
└──────────────────────────────────────┘
```

### 1.2 翻看和选择

- **左右箭头翻页**，不是 tab（草稿可能很长，tab 不合适）
- 翻页时正文区域整体替换，带轻微淡入动画
- 当前展示的稿件高亮显示稿件编号
- 每稿底部小字显示该稿的 生成时间 + 使用的模型 + temperature
- 用户可以在任意一稿的正文上直接编辑（编辑后该稿标记"已修改"）

### 1.3 定稿流程

- 用户在任意一稿上点 [✅ 定稿] → 确认的是**当前正在看的这一稿**
- 前端发送确认请求时携带 draft_id（如 ch038_draft_B.md）
- 后端校验 draft_id 对应的文件存在后执行 confirm
- 确认后所有其他草稿自动清理

### 1.4 草稿数量限制

- Phase 1 不限制数量，但 UI 建议最多 5 稿
  （超过 5 稿时 [换一版] 按钮旁提示："已有 5 个版本，建议先从中选一个定稿"）
- 每次生成消耗 API 额度，用户自行决定

### 1.5 崩溃恢复

- 重新打开写作界面时，前端检查 `.drafts/` 目录
- 若存在草稿 → 强制进入草稿浏览模式，显示所有已有草稿
- 用户必须选择 [定稿某一个] 或 [全部丢弃] 才能回到正常写作状态
- 不允许在有未处理草稿时生成新内容

## 2. 用户怎么知道 AI "用了" 设定

### 2.1 问题

用户添加了角色设定和剧情笔记，但续写后无法感知 AI 是否参考了这些内容。
Context 可视化面板太技术化（P0/P1/P2...token 数），普通用户不会打开。

### 2.2 方案：轻量参考摘要（续写结果下方）

每次续写完成后，在正文下方用小字灰色显示一行摘要：

```
┌──────────────────────────────────────┐
│                                      │
│          （续写正文）                  │
│                                      │
│ ─────────────────────────────────── │
│ 📎 本次参考了：Connor 设定 · 圣锚星系  │
│    世界观 · 3条剧情笔记 · 1条底线      │
│                          [详情 ▼]    │
└──────────────────────────────────────┘
```

**数据来源**：后端 `assemble_context()` 完成后，返回一个 `context_summary` 对象：

```json
{
  "characters_used": ["Connor Ellis", "Colin"],
  "worldbuilding_used": ["圣锚星系.md"],
  "facts_injected": 3,
  "facts_as_focus": ["那句没说完的话"],
  "pinned_count": 1,
  "rag_chunks_retrieved": 2,
  "total_input_tokens": 12450,
  "truncated_layers": []
}
```

### 2.3 点击 [详情] 展开

```
📎 本次 AI 参考了以下内容：

  👤 角色设定（2个）
     Connor Ellis（完整注入）
     Colin（完整注入）

  🌍 世界观（1个）
     圣锚星系（智能检索匹配）

  📝 剧情笔记（3条）
     ★★★ "Connor 和 Colin 目前处于热恋期" — 生效中
     🔴  "那句没说完的话" — 本章重点推进
     ★★  "圣所计划的压力" — 生效中

  🚧 底线（1条生效）

  ⚠️ 被精简的内容：无
     （如果有被截断的设定，这里会显示警告）
```

### 2.4 被截断时的主动提醒

如果 Context Assembler 截断了角色设定（P5 层被裁剪），正文下方摘要变为黄色警告：

```
⚠️ 本次有设定被精简：龙套角色「陈律师」的设定未注入（超出容量）
   [查看详情] [调整必带角色]
```

这样用户不需要理解 token 和 P0-P5，但能直观感知"AI 是不是忘了我的设定"。

## 3. 确认章节的完整交互流（嗑粮模式）

### 3.1 流程图

```
用户看完续写结果
      ↓
  满意吗？
  ├── 不满意 → [换一版] 或 [丢弃] → 回到写作
  └── 满意 → 点 [✅ 定稿]
                ↓
        ┌─────────────────────────┐
        │  确认第38章？            │
        │                         │
        │  (正文预览前200字...)     │
        │                         │
        │  [确认定稿]  [取消]       │
        └─────────────────────────┘
                ↓ 确认
        后端执行 confirm_chapter：
        写入 chapters/main/ + 更新 state + 向量化入队
                ↓
        ┌─────────────────────────┐
        │ ✅ 第38章已定稿！         │
        │                         │
        │ 💡 要不要记几条剧情笔记？ │
        │ AI 可以帮你从这章提取：   │
        │                         │
        │ [✨ 让AI提取] [自己记]    │
        │ [跳过，继续写下一章 →]    │
        └─────────────────────────┘
                ↓
        选"让AI提取"：
          → 调用轻量事实提取 API
          → 展示提取结果列表，每条可勾选
          → [保存选中的] → 写入 facts.jsonl
          → 进入下一章写作

        选"跳过"：
          → 直接进入下一章写作
          → 下次不再提醒（可在设置中重新开启）
```

### 3.2 关键设计决策

- **确认是一键操作 + 二次确认弹窗**，不需要填任何表单
- **facts 提取是可选的**，不阻断写作流（嗑粮优先原则）
- **"跳过"是受尊重的选择**，不反复弹窗骚扰
- 定稿后输入框自动聚焦，chapter_focus 选择器刷新，进入下一章状态
- 如果用户勾选"以后都不提醒"，定稿后直接跳到下一章

### 3.3 定稿前能不能编辑

**能。** 用户可以直接在正文区域编辑草稿文字，编辑后定稿。
编辑时草稿标记为"已修改"，定稿时用修改后的内容写入。
`provenance` 字段从 `ai` 改为 `mixed`。

## 4. 必带角色（core_always_include）在哪设置

### 4.1 问题

`core_always_include` 决定哪些角色的设定每次续写都完整注入，
是"不崩人设"最核心的功能。但 PRD 里：
- AU Settings 页面没有这个入口
- 设定集页面没有标记为"必带"的操作
- 用户不知道这个功能存在

### 4.2 方案：在设定集页面的角色卡片上操作

设定集（AU Lore）的角色列表中，每个角色卡片增加一个"必带"标记：

```
┌──────────────────────────────────────┐
│ 📁 角色设定 (3)                       │
│                                      │
│ ┌────────────────────────────────┐   │
│ │ 📌 Connor Ellis    主角         │   │
│ │ 别名：皇帝陛下, Connor          │   │
│ │ 来源：原作 Connor               │   │
│ │ [📌 必带] ← 已开启，蓝色高亮    │   │
│ └────────────────────────────────┘   │
│                                      │
│ ┌────────────────────────────────┐   │
│ │ 📌 Colin           主角         │   │
│ │ [📌 必带] ← 已开启              │   │
│ └────────────────────────────────┘   │
│                                      │
│ ┌────────────────────────────────┐   │
│ │    陈律师           龙套         │   │
│ │ [📌 设为必带] ← 未开启，灰色    │   │
│ └────────────────────────────────┘   │
│                                      │
│ ℹ️ "必带"角色的设定每次续写都会       │
│    完整提供给 AI，确保不会崩人设。     │
│    建议只设主角（1-3个），设太多       │
│    会占用 AI 的记忆空间。             │
└──────────────────────────────────────┘
```

### 4.3 操作方式

- 点击 [📌 设为必带] → 角色加入 core_always_include
- 再次点击 → 取消必带
- 设为必带时自动校验：
  - 如果该角色设定里没有 `## 核心限制` 段落，弹出提示：
    "建议在设定中添加「核心限制」段落（2-3句话概括最重要的人设），
     当 AI 记忆空间不足时，至少会保留这几句话。"
- 推荐不超过 3 个必带角色（超过 3 个时提示但不阻止）

### 4.4 新建 AU 时的引导关联

新建 AU 添加了角色后（通过 AI 配置/导入/手动创建），
如果 core_always_include 还是空的，在设定集页面顶部显示一次性引导：

```
💡 你还没有设置"必带角色"。
   把主角设为必带，AI 每次续写都会记住他们的人设。
   [去设置]  [稍后]
```

## 5. Fandom 设定模式 vs AU 设定模式的详细区别

### 5.1 定位对比

| 维度 | Fandom 设定模式 | AU 设定模式 |
|------|---------------|------------|
| 入口 | Fandom 设定库页面内嵌 | 写作界面顶部 [🔧 改设定] 切换 |
| 用途 | 整理原作角色分析、世界观笔记 | 配置本篇的角色/世界观/剧情笔记/底线 |
| AI 角色 | "角色分析助手" | "设定管理助手" |
| 可操作文件 | core_characters/*.md, worldbuilding/*.md | characters/*.md, worldbuilding/*.md |
| 可操作数据 | 无（Fandom无facts/底线） | facts, pinned_context, writing_style, core_includes |
| AI 参考上下文 | 无（Fandom层是源头） | Fandom层DNA摘要 + AU现有设定 |
| tool 数量 | 4个（仅文件CRUD） | 9个（文件+facts+底线+文风+核心绑定） |
| 对话历史 | 内存，关闭Fandom页清空 | 内存，关闭AU或切回写作模式不清空（切回后保留） |

### 5.2 Fandom 设定模式的 tool 定义

只有4个（比 AU 的9个少）：

```
create_core_character_file   → POST /api/v1/lore (fandom层, category=core_characters)
modify_core_character_file   → PUT /api/v1/lore (fandom层)
create_worldbuilding_file    → POST /api/v1/lore (fandom层, category=worldbuilding)
modify_worldbuilding_file    → PUT /api/v1/lore (fandom层)
```

没有 add_fact、add_pinned_context、update_writing_style、update_core_includes
——因为这些都是 AU 级概念，Fandom 层不存在。

### 5.3 Fandom AI 的行为差异

Fandom AI 更像一个"角色分析伙伴"：

```
用户："帮我分析一下 Connor 这个角色"
Fandom AI："从原作来看，Connor 最核心的特质是……（分析）
         我建议创建以下设定文件来记录你的理解：
         [create_core_character_file: Connor.md]"

用户：(粘贴3000字角色分析)
Fandom AI："我帮你整理了结构，标注了核心人格段落：
         [create_core_character_file: Connor.md，保留原文]"
```

AU AI 更像一个"设定配置助手"：

```
用户："这个AU的Connor是星际皇帝"
AU AI：(读取Fandom层Connor的DNA)
      "基于你对 Connor 的人格分析，我为星际皇室 AU 适配了设定：
       [create_character_file: Connor Ellis]
       [add_fact: Connor和Colin的关系]
       [add_pinned_context: 当前处于热恋期]"
```

### 5.4 UI 布局差异

**Fandom 设定模式** — 嵌入在设定库页面内，不是全屏：

```
┌──────────────────────────────────────────┐
│ ← 底特律变人 · 原作资料                   │
├──────────────────────────────────────────┤
│ 🤖 AI 助手                               │
│ ┌──────────────────────────────────────┐ │
│ │ （对话历史区域，较小）                  │ │
│ └──────────────────────────────────────┘ │
│ ┌──────────────────────────────────────┐ │
│ │ 粘贴角色分析或描述...        [发送]   │ │
│ └──────────────────────────────────────┘ │
│                                          │
│ ── 文件列表 ──                            │
│ 📁 角色 (2)                               │
│   Connor.md · Hank.md                     │
│ 📁 世界观 (1)                             │
│   仿生人社会.md                            │
└──────────────────────────────────────────┘
```

**AU 设定模式** — 在写作界面内，全屏：

```
┌──────────────────────────────────────────┐
│ [✍️ 写文] [📋 改设定]      ← 当前高亮     │
├──────────────────────────────────────────┤
│                                          │
│  （AI 对话 + 确认卡片区域，和写作正文共用  │
│   同一块大面积空间）                       │
│                                          │
├──────────────────────────────────────────┤
│ 描述你想要的设定变更...          [发送]    │
└──────────────────────────────────────────┘
```

### 5.5 为什么分开而不是统一

- Fandom 设定是"知识库维护"，频率低，不需要和写作流程耦合
- AU 设定是"写作过程中随时调整"，必须和写作界面紧密集成
- 操作范围完全不同（D-0025），合并会导致 AI 操作越界风险
- System prompt 不同：Fandom AI 鼓励深度分析，AU AI 侧重快速配置
