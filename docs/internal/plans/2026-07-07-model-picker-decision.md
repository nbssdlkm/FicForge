# 决策材料：要不要做「供应商主导的模型选择器」

> **已拍板（用户，2026-07-07）：做，按方案 B（内置供应商清单 + 推荐模型下拉 + 一键拉取列表 + 手填三层）。**
> 硬性要求：**必须保留「自定义供应商」和「自定义模型」**（用户可自行添加任意 OpenAI 兼容地址与模型名，且自定义模型可手填窗口大小）。
> 交互设计参照开源应用 Kelivo 的模型选择器（调研笔记见同目录 `2026-07-07-kelivo-model-picker-notes.md`）。

> 配套调研数据：`2026-07-07-model-landscape-research.md`（同目录）。
> 用大白话写，按「背景 / 影响 / 后果 / 为何推荐 / 为何要你拍板」五段。

## 一、背景（现在是什么情况）

现在用户配模型要自己**手打**三样东西：接口地址、API key、模型名字。模型名字打错一个字母，报错只有一句看不懂的 404。app 里有一张内置的「模型名字 → 窗口大小」对照表，用来算每次续写能塞多少上下文——这张表的数据停在半年前：表里最重要的条目 `deepseek-chat`，官方已宣布 **7 月 24 日起直接报错停用**；现在主流模型的窗口已经是 256k 到 1M，表里还按 32k、64k 在算。对照表查不到的模型一律按 32k 保守处理——用 1M 窗口的模型会浪费掉九成七的容量。

（预算计算读错配置的 bug 本身——H4——已经在修，和这个决策无关；这里说的是**数据过时**和**用户体验**两件事。）

## 二、影响（这个决策改变什么）

做「供应商主导的选择器」= 设置里先选供应商（DeepSeek / Kimi / 智谱 / 硅基流动……），app 自动填好接口地址，然后从一个**推荐模型下拉**里选（每家 2-3 个，标着「旗舰 / 便宜 / 长上下文」），窗口大小自动带出来。不想用推荐的，仍可点「从 API 拉取完整列表」或者手填。

不做 = 维持手打模型名，只把内置对照表的数据刷新一遍（这个刷新无论如何都会做，因为它是数据错误不是产品选择）。

## 三、后果（各选项的代价）

- **方案 A：只刷新数据，不做选择器**。零新代码。但半年后数据又过时（这次调研证明：国内厂商 6-8 周一次大版本，半年约半数条目失效，而且有的失效方式是「旧名字直接报错」）。用户仍要手打模型名。
- **方案 B：内置供应商清单 + 推荐下拉 + 拉取列表 + 手填（三层兜底）**。业界同类产品（Cherry Studio、LobeChat、SillyTavern）全是这个结构，没有一家靠纯静态表。代价：一次性开发（估计中等工作量：一个清单数据文件 + 设置页下拉改造），清单数据仍要随发版更新，但因为有「拉取列表 + 手填」兜底，过时只是「推荐不够新」，不会「用不了」。
- **方案 C：B 的基础上把清单放到网上（app 启动时拉最新，拉不到用本地缓存）**。清单更新不用发版。代价：多一个网络依赖 + 国内可达性要处理（可用 CDN 解决），首版可以不做、留扩展点。

## 四、为何推荐 B（首版），留 C 的扩展点

- 你的用户是不太懂技术的写手，「选供应商 → 选推荐模型」比「手打模型名」少掉最大的一类配置失败。
- 窗口大小数据借选择器一并带出，1M 模型不再被当 32k 用——这直接放大每次续写的记忆容量，是记忆栈体验的隐形地板。
- 三层兜底意味着清单过时不致命，维护压力可控；C 的远程清单等真实用户量说话再上。
- 清单数据文件做成单一真相源（供应商、地址、推荐模型、窗口、标签都在一个文件里），改数据不用动逻辑代码——符合项目一贯的架构规矩。

## 五、为何要你拍板

这是**产品功能取舍**不是 bug 修复：要不要把「选模型」做成引导式体验、设置页要改布局、清单里放哪几家供应商排什么顺序（这直接影响新用户第一次配置的成功率），都是产品决定。另外方案 B 有真实的开发和维护成本，值不值得花在现在这个阶段（相对于其他 P1/P2 修复），应该由你定优先级。

## 附：无论怎么拍都会做的两件事

1. **MODEL_CONTEXT_MAP 数据刷新**（含 fuzzy 匹配修复：`org/模型名` 形态的 id 现在全部落到 32k 兜底）——数据错误，直接修。
2. **提醒**：你自己的 deepseek 测试配置用的是 `deepseek-chat`，7 月 24 日后会直接报错，需要换成 `deepseek-v4-flash`（或到时官方指定的新名字）。
